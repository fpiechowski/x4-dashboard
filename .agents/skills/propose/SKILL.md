# Propose

## Workflow

1. Start from the new idea, request, or issue context.
2. Read only the roadmap, issue context, and repo areas needed to understand where the work belongs.
3. Decide whether to:
   - update an existing issue
   - create a new issue
   - update `ROADMAP.md`
   - do more than one of the above when the planning picture changes
4. Place the work in the most appropriate milestone or explicitly leave it unscheduled when the timing is not ready.
5. Write a concise planning shape with:
   - summary
   - goal
   - scope
   - acceptance criteria
   - milestone or release placement when it is clear
6. Stop at planning output. Do not begin implementation.

## Output

- What was proposed
- Where it now lives
- Which issue or roadmap item was created or updated
- Why that milestone or backlog placement was chosen

## Guardrails

- Do not implement product code in this workflow.
- Prefer updating an existing issue instead of creating duplicates.
- Keep the result actionable enough that `refine` can take over later.
